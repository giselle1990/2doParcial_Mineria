Cloud Provider Analytics:
Pipeline Lambda con PySpark + Astra DB

Implementación de un MVP técnico para procesar datos de consumo cloud mediante ingesta batch y streaming, arquitectura medallón, reglas de calidad, mart analítico FinOps y capa de serving en Astra DB.

MotorPySpark 3.5.0
ModeloLambda + Medallón
StorageParquet / Datalake
ServingAstra DB
Grano Goldorg_id + fecha + servicio
SalidaMart FinOps

1. Objetivo del MVP

El proyecto demuestra un flujo de datos completo desde archivos crudos hasta una base de consulta optimizada.

1

Procesar datos de entrada

Ingestar maestros en formato CSV y eventos de uso en formato JSONL.

2

Aplicar calidad y trazabilidad

Tipificar, limpiar, enriquecer, deduplicar y separar registros inválidos en quarantine.

3

Publicar métricas consultables

Generar un mart FinOps y cargarlo en Astra DB para consultas por organización, fecha y servicio.

3. Arquitectura general

El pipeline combina procesamiento batch para maestros y streaming para eventos.

LandingCSV y JSONL sin transformar
BronzeDatos tipificados con metadata
SilverLimpieza, reglas y enriquecimiento
GoldAgregación analítica FinOps
ServingAstra DB / Cassandra
ConsultasConsumo y ranking de costos

Patrón Lambda

La solución incluye dos caminos de ingesta: batch para datos maestros y streaming estructurado para eventos de uso. Ambos caminos convergen en capas analíticas posteriores.

Arquitectura medallón

Bronze conserva trazabilidad; Silver mejora calidad y consistencia; Gold produce métricas agregadas listas para consulta y visualización.

Se procesan eventos con Structured Streaming, watermark de 10 minutos, checkpoint y deduplicación por event_id.

Se agrupan los eventos válidos por organización, fecha y servicio para generar métricas de costo y uso.

df_gold_finops = (df_silver_events
  .groupBy("org_id", "event_date", "service_norm")
  .agg(
    F.sum("cost_usd_increment").alias("daily_cost_usd"),
    F.count("*").alias("requests"),
    F.sum("value").alias("total_value"),

8. Serving en Astra DB

El diseño de Cassandra/Astra DB se realiza con enfoque query-first.

Modelo CQL

CREATE TABLE IF NOT EXISTS default_keyspace.org_daily_usage_by_service (
  org_id TEXT,
  usage_date DATE,
  service TEXT,
  daily_cost_usd DOUBLE,
  requests BIGINT,
  total_value DOUBLE,
  cpu_hours DOUBLE,
  storage_gb_hours DOUBLE,
  genai_tokens DOUBLE,
  carbon_kg DOUBLE,
  anomaly_flag TEXT,
  PRIMARY KEY ((org_id, usage_date), service)
);

Justificación de clave

DecisiónMotivo
(org_id, usage_date) como partition keyPermite consultar consumo por organización y fecha.
service como clustering columnOrdena y diferencia servicios dentro de la misma partición.
Modelo desnormalizadoEvita joins en Cassandra y favorece consultas directas.

9. Validación de idempotencia

El pipeline puede re-ejecutarse sin duplicar registros.

Batch

Se aplica unionByName y dropDuplicates por clave de negocio. Al reprocesar el mismo CSV, el conteo final no aumenta.

Streaming

Se usa checkpoint, watermark y dropDuplicates(["event_id"]) para controlar relecturas y duplicados.

Serving

En Data API se usa replace_one con upsert=True. En CQL, la primary key evita duplicados lógicos.
